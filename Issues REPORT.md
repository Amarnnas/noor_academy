## 🟡 إصلاحات "جزئية" — تم تعديل الـ API لكن الواجهة/Dashboard لم تُربط بعد

### 🟡 2.1 — الطلبات (Orders) الآن تُحفظ في Firestore... لكن لوحة التحكم لا تعرضها
**`src/app/api/orders/route.ts`**: أصبح يستدعي `createOrder()` من `firestore.ts` فعلياً ✅ — هذا تقدّم حقيقي ومهم.

لكن **`src/app/(dashboard)/dashboard/orders/page.tsx`** ما زال:
```ts
import { initialOrders } from "@/data/orders";
```
أي أن المشرف عندما يفتح "الطلبات" في لوحة التحكم **سيرى بيانات ثابتة وهمية فقط، ولن يرى أي طلب حقيقي جديد** تم إنشاؤه عبر الموقع، رغم أنه محفوظ بالفعل في Firestore.

**المطلوب لإكمال الإصلاح:** تعديل صفحة `dashboard/orders` لاستخدام `getAllOrders()` من `firestore.ts` (مع تحويلها لصفحة/جزء `async` أو جلب البيانات عبر `useEffect` + `getDocs`)، وربط أزرار "تحديث الحالة" بـ `updateOrder()`.

### 🟡 2.2 — رسائل التواصل الآن تُحفظ في Firestore... لكن لوحة التحكم تعرض بيانات Mock ثابتة بالكامل
**`src/app/api/contact/route.ts`**: يستدعي `createMessage()` فعلياً ✅.

لكن **`src/app/(dashboard)/dashboard/messages/page.tsx`** يحتوي مصفوفة ثابتة مكتوبة يدوياً بالكود نفسه:
```ts
const messages: ContactMessage[] = [
  { id: "1", name: "فهد العنزي", ... },
  { id: "2", name: "منيرة الدوسري", ... },
  { id: "3", name: "خالد الزهراني", ... },
];
```
لا يوجد أي استدعاء لـ `getAllMessages()` الموجودة وجاهزة في `firestore.ts`. أي أن **رسائل "تواصل معنا" الحقيقية تُحفظ الآن في قاعدة البيانات، لكن لا توجد أي واجهة لقراءتها** — وهي عملياً مخفية عن المشرف تماماً (نفس النتيجة العملية كالسابق: المشرف لن يرى رسائل الزوار، لكن السبب تغيّر من "تضيع" إلى "محفوظة لكن غير معروضة").

**المطلوب:** ربط هذه الصفحة بـ `getAllMessages()` + `markMessageRead()` + `deleteMessage()`.

### 🟡 1.5 — Rate limiting على OTP: تحسين حقيقي، لكن بقيود بيئة serverless
**`src/app/api/auth/otp-verify/route.ts`**: أُضيف نظام 5 محاولات/10 دقائق ✅ — تحسين كبير وصحيح من حيث المنطق.

⚠️ **لكن** نفس ملاحظة `user-store.ts` (البند 2.4 أدناه) تنطبق هنا: العداد مخزّن في `Map` في الذاكرة (`const attempts = new Map(...)`). على Vercel (serverless):
- كل "نسخة تنفيذ" (instance) قد تملك `Map` خاصة بها.
- عند Cold Start تُعاد تهيئة `Map` فتُمسح كل المحاولات السابقة.

**النتيجة:** الحماية تعمل بشكل جيد ضد محاولات سريعة متتالية على نفس instance، لكنها **ليست ضماناً كاملاً** ضد مهاجم منظّم (لو وزّع الطلبات على فترات/استبدال instances). هذا تحسين مهم لكنه ليس الحل النهائي.

**المطلوب لحل جذري:** استخدام تخزين خارجي مشترك بين كل الـ instances مثل **Vercel KV / Upstash Redis** لعدّاد المحاولات (ونفس الكلام ينطبق على `otp-store.ts` نفسه أصلاً، وعلى `user-store.ts`).

**الأهم الآن (بالترتيب):**

1. **1.3 + 1.2 معاً**: هما السبب الجذري الحقيقي — يجب نقل كل عمليات إدارة `admins/teachers/students` إلى API routes في السيرفر (مع `getServerSession` + فحص الصلاحية) باستخدام **Firebase Admin SDK**، وتشفير كل كلمات المرور بـ `bcrypt` قبل التخزين/المقارنة. بدون هذا، الفحوصات الأخرى (rate limiting، إخفاء OTP...) لن تحمي قاعدة البيانات نفسها من القراءة/الكتابة المباشرة إن كانت Firestore Rules مفتوحة.
2. **2.4**: نقل `user-store.ts` إلى Firestore (collection `students`) — وإلا فإن التسجيل اليدوي للطلاب (وليس عبر Google) سيبقى عشوائي العمل على الاستضافة الحالية.
3. **2.1 / 2.2 (إكمال الربط)**: تعديل صفحتي `dashboard/orders` و`dashboard/messages` لقراءة Firestore الحقيقي — هذا الآن أهم لأن البيانات أصبحت تُحفظ فعلياً ولكنها "مخفية" عن المشرف، وهذا قد يُعطي انطباعاً خاطئاً بأن النظام لا يستقبل أي طلبات/رسائل.
4. **2.3**: ربط إدارة الدورات/المدربين بـ Firestore (نفس النمط المستخدم في صفحة admins، لكن عبر API محمي بالسيرفر هذه المرة لا client SDK مباشر — تجنباً لتكرار مشكلة 1.3).
5. **2.5**: إكمال تدفق استعادة كلمة المرور بخطوة فعلية لتحديث الباسورد (مشفّراً).
6. **2.6**: استبدال الكتابة على نظام الملفات بـ Firebase Storage أو حل تخزين سحابي مناسب لـ serverless.
7. **3.1 و4.1**: مراجعة صلاحيات teacher على مستوى الخادم، والتحقق من صحة إصدارات `package.json` عبر `npm install` فعلي.
